# خطة تنفيذ الموجة الأولى — Wave 1 Implementation Plan

## الفلسفة
- 🎯 **نبني على البنية القائمة، لا نعيد اختراعها.** كل مديول جديد يستفيد من: المستخدمون، الفروع، RLS، `canAccessModule()`، `route()`، `acct_inventory_items` (للوصفات)، `acct_customers` (للعملاء)، `ops_orders` (للطلبات).
- 🛡️ **صفر كسر لأي شيء شغّال.** كل جدول جديد `IF NOT EXISTS`, كل route جديد يُضاف في نهاية سلسلة `else if`, كل مديول خلف `canAccessModule()` — test_admin فقط.
- 📦 **كل مديول = 3 commits** (Schema → DAL → Screens) للمراجعة السهلة والاسترجاع الآمن.

## ترتيب التنفيذ (بالأولوية والتبعية)

### 🥇 المديول 24 — المنيو والوصفات (نبدأ الآن)
**لماذا أولًا؟** أساس POS وCRM (تفضيلات العميل)، ويعطي قيمة فورية حتى قبل بناء POS: معرفة تكلفة كل صنف وهامش الربح، منع بيع صنف نفدت مكوناته، تسعير علمي بدل تخميني.

**الجداول (5):**
- `menu_categories` — الفئات (وجبات، مشروبات، حلويات)
- `menu_items` — الأصناف بأسعارها الأساسية
- `menu_item_recipes` — Bill of Materials (المكونات من `acct_inventory_items`)
- `menu_channel_prices` — أسعار مختلفة للقنوات (dine_in / takeaway / delivery)
- `menu_settings` — إعدادات مركزية (هامش الربح المستهدف الافتراضي، عرض السعر شامل الضريبة أم بدونها)

**الشاشات:**
- Dashboard — KPIs (عدد الأصناف، متوسط الهامش، الأصناف الأدنى ربحية)
- الفئات — قائمة + CRUD + إعادة ترتيب
- الأصناف — قائمة بشارة الهامش (أخضر/أصفر/أحمر)
- تفاصيل الصنف — البيانات + محرر الوصفة + محاكي تسعير + أسعار القنوات

**التكاملات:**
- ⬅️ `acct_inventory_items.unit_cost` — تكلفة كل مكون (موجود)
- ⬅️ `acct_settings.vat_rate` — احتساب السعر شامل الضريبة (موجود)
- ➡️ POS مستقبلًا: يقرأ المنيو ويخصم المكونات عند البيع
- ➡️ CRM مستقبلًا: يحفظ تفضيلات العميل من المنيو

**Feature Flag:** إضافة `'menu'` إلى `PREVIEW_MODULES`

**تقدير الجهد:** جلسة عمل واحدة (SQL + DAL + 4 شاشات).

### 🥈 المديول 25 — إدارة العملاء والولاء (CRM)
**لماذا ثانيًا؟** يعمل مستقلًا (لا يحتاج POS)، يمكن البدء بتسجيل العملاء يدويًا من الآن، ويصبح أقوى بعد POS.

**الجداول (5):**
- `crm_customers` — قاعدة العملاء (اسم، جوال، عنوان، تفضيلات، حساسيات، حالة)
- `crm_customer_addresses` — عناوين متعددة لكل عميل
- `loyalty_accounts` — حساب النقاط (مستوى، رصيد، تاريخ الترقية)
- `loyalty_transactions` — كل حركة نقاط (اكتساب/استبدال)
- `crm_complaints` — الشكاوى بحالاتها

**الشاشات:**
- Dashboard — KPIs (عدد العملاء، النشطون، الخاملون، القيمة الحياتية CLV)
- قائمة العملاء — بحث وتصفية بالشريحة
- تفصيل العميل — بياناته + تفضيلاته + تاريخ الطلبات (فارغ لحد ما POS يجهز) + رصيد النقاط + شكاواه
- الشكاوى — قائمة + إنشاء + متابعة + إغلاق

**التكاملات:**
- ⬅️ `ops_orders` (لاحقًا) — ربط الطلب بعميل
- ➡️ POS مستقبلًا: تسجيل بيع للعميل + منح نقاط
- ➡️ الجودة: شكوى قد تولد إجراء تصحيحي

**Feature Flag:** إضافة `'crm'` إلى `PREVIEW_MODULES`

**تقدير الجهد:** جلسة عمل واحدة.

### 🥉 المديول 23 — نقاط البيع POS
**لماذا ثالثًا؟** يعتمد على المنيو (الأصناف والوصفات) وCRM (اختياريًا ربط بعميل). يجب أن يكون الاثنان جاهزين قبل بنائه.

**الجداول (6):**
- `pos_terminals` — تعريف الأجهزة في كل فرع
- `pos_sessions` — جلسات الكاشير (فتح/إغلاق صندوق)
- `pos_transactions` — كل معاملة بيع
- `pos_transaction_items` — أصناف الطلب
- `pos_payment_splits` — تفاصيل الدفع المختلط
- `pos_cash_deposits` — إيداعات النقد

**الميزات الحرجة:**
- شاشة كاشير سريعة بلمس
- Offline mode (حفظ محلي ومزامنة لاحقة)
- QR ZATCA على الإيصال
- خصم مكونات المخزون تلقائيًا عبر الوصفة
- قيد محاسبي فوري (نقدية / إيرادات + VAT)
- إغلاق وردية بجرد وتوثيق فروقات

**Feature Flag:** إضافة `'pos'` إلى `PREVIEW_MODULES`

**تقدير الجهد:** جلستان (SQL + Core → Full offline + printer + KDS).

## القواعد الثابتة (تتكرر لكل مديول)

| القاعدة | التنفيذ |
|---|---|
| SQL idempotent | `IF NOT EXISTS` + `DROP POLICY IF EXISTS` + `ON CONFLICT DO NOTHING` |
| RLS صارم | استخدام `is_operations_manager()` / `is_finance_manager()` / role checks |
| Feature flag | `PREVIEW_MODULES.push('module_name')` + gate في route + coming-soon page |
| قرارات مؤسسية | `// DECISION:` comment لكل قرار حاسم |
| فحص syntax | Node `new Function()` على كل script block قبل commit |
| Version bump | `APP_VERSION` + `SW_VERSION` معًا |
| Commit واضح | مرحلة واحدة = commit واحد بوصف مفصل |
| Push فوري | بعد كل commit ناجح |
